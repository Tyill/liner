# Семантика старта хранилища и сбоев

## Клиент и внутренние задачи

**Путь клиента** написан так, чтобы ошибки хранилища не приводили к панике: операции, затрагивающие бэкенд, могут завершиться ошибкой (таймауты Redis, SQLite `SQLITE_BUSY` после настроенного ожидания, ошибки I/O). Такие сбои проявляются как ошибки API или контролируемое завершение, а не как аварийный выход всего процесса.

**Задачи listener и sender** при старте `run` открывают собственный дескриптор хранилища (тот же `unique_name` и `StoreBackend`, что у клиента). На этом пути открытие хранилища и разрешение ключа топика используют fail-fast: если хранилище не открыть или ранний запрос падает, процесс может **паниковать** с явным сообщением. Это намеренно.

## Почему старт listener строже

После успешной инициализации клиента **редко**, чтобы открытие хранилища listener’ом не удалось, пока у клиента оно ещё работает. Обе стороны используют один логический ресурс (URL Redis или путь SQLite). Расхождение обычно означает что-то ненормальное: гонку (права, место на диске), эксклюзивную блокировку SQLite другим процессом или «дрожание» инфраструктуры между двумя открытиями.

В таких случаях наполовину запущенный брокер (сеть есть, хранилище «упало» только на стороне listener) сложнее анализировать, чем немедленный отказ. Библиотека исходит из того: **если вы вызываете `run`, хранилище должно быть доступно для старта listener/sender**, а не только для объекта клиента, который вы создали первым.

## Практический вывод

- Ожидайте **аккуратную** обработку ошибок хранилища в **текущих операциях клиента**.
- Ожидайте **fail-fast** (возможная паника), если хранилище недоступно при **запуске listener или sender внутри `run`**, даже если клиент был создан успешно.

Если нужно, чтобы процесс оставался жив при втором неудачном открытии, потребовалось бы сознательное изменение API (проброс `Result` из внутреннего старта вместо `expect`); сейчас так не сделано.
