# Поведение: топики, доставка и ошибки

Здесь описано **что делает библиотека** с точки зрения приложения: именование, маршрутизация, семантика доставки и режимы сбоев. **Формат байтов по TCP** здесь не задаётся.

Имена ключей Redis / таблиц SQLite (для операторов) — [routing-and-store-layout.md](routing-and-store-layout.md). **stderr, коды возврата и паники** — [errors-and-logging.md](errors-and-logging.md). **Офлайн-очереди, тайминг переподключения и `number_mess`** — [offline-delivery-and-message-numbers.md](offline-delivery-and-message-numbers.md).

## Топики

### Исходный топик (собственный топик клиента)

При создании вы передаёте **строку топика** и **TCP-адрес привязки** (`localhost`). После успешного **`run`** клиент **регистрирует** эту пару в общем хранилище: другие процессы узнают, *«этот топик обслуживается по этому TCP-адресу под этим `unique_name`»*.

Этот топик — **исходный топик** клиента. API **запрещает** отправку на него или подписку на него как на удалённого пира — такие вызовы завершаются с сообщением об ошибке.

### Целевые топики и адреса

**`send_to(topic, …)`** и **`send_all(topic, …)`** используют `topic` как **логическое имя**. Клиент ищет **TCP-адреса**, зарегистрированные для этого топика в хранилище.

- Если **адресов нет**, отправка не удаётся («not found addr for topic …»).
- Если **несколько адресов** (несколько клиентов с одним именем топика), **`send_to`** на каждый вызов выбирает **один адрес** по **round-robin** по кэшированному списку. **`send_all`** отправляет тот же payload **по одному разу на каждый зарегистрированный адрес** (широковещание всем репликам в списке для топика).

Клиент **кэширует** адреса после первого успешного поиска. Пока пиры работают, **внутренний канал** (`__#internal_channel`) обновляет кэш при connect, disconnect, subscribe и unsubscribe (см. [using-the-api.md](using-the-api.md)). Вызывайте **`refresh_address_topic(topic)`**, когда нужно принудительно перечитать store — например после смены порта без корректного disconnect, при подписке до `run` или если отправитель не работал в момент регистрации пира.

### Внутренний канал (обнаружение пиров)

При **`run`** каждый клиент автоматически подписывается на **`__#internal_channel`**. Библиотека обменивается JSON-событиями между пирами и обновляет кэш адресов **без** вызова прикладного колбэка приёма. В типичном случае отправителю не нужен ручной **`refresh_address_topic`** после **`run`** или **`subscribe`** пира в рантайме. Крайние случаи — в [using-the-api.md](using-the-api.md) (подписка до `run`, гонки, устаревший кэш).

### Подписки (сторона приёма)

**`subscribe(topic)`** регистрирует интерес получать **прикладные полезные нагрузки**, чьи **метаданные по проводу** относятся к внутреннему **ключу топика** этого имени. В колбэк приёма попадают только топики, на которые вы подписались; остальные отбрасываются (с отладочным сообщением). **`unsubscribe`** снимает эту привязку.

Обычно **`subscribe`** вызывают для топиков, которые хотите слушать **дополнительно** к исходному топику (например широковещательные каналы). Подписки можно зарегистрировать **до** или **после** **`run`**; если до — они применяются при старте listener.

## Доставка

### Транспорт

После того как маршрутизация разрешила **TCP-адрес**, встроенный поток **sender** открывает **обычное TCP**-соединение к **listener** пира и передаёт кадрированные сообщения. **TLS** в самой библиотеке **нет** (см. [security-defaults.md](security-defaults.md)).

### At-least-once и best-effort

API отправки экспортируют **`at_least_once_delivery`** (C: `BOOL`). При **включении** стек может **сохранять** сообщения, которые ещё не доставлены, и **повторять** после переподключений, используя **номера сообщений** по каналу и сохранённые позиции **ack**, чтобы получатель не применял дубликаты. При **выключении** отправки **best-effort**: при сбоях или офлайне пира данные могут **теряться** без записи в хранилище.

Точные правила, таймеры и дедупликация — в [offline-delivery-and-message-numbers.md](offline-delivery-and-message-numbers.md).

### Упорядочивание

Библиотека стремится к **FIFO на логический канал отправитель–получатель** (`connection_key`) с учётом сбоев и повторов. **Глобального** полного порядка по всем топикам или всем клиентам **нет**.

### Контракт колбэка

Колбэк приёма получает **указатели, действительные только на время вызова**; скопируйте байты, если они нужны позже (см. [using-the-api.md](using-the-api.md)).

## Ошибки (поведенческое резюме)

- **Создание:** если хранилище не открылось или параметры неверны, в Rust — **`None`**, в C — **`NULL`**. Иначе при простом сбое открытия stderr часто пуст (см. [errors-and-logging.md](errors-and-logging.md)).
- **`run`:** возвращает **`false`**, если регистрация топика не удалась, адрес привязки неверен или TCP bind не удался. Возвращает **`true`**, если клиент **уже** был в running (идемпотентно). Внутренний старт listener/sender всё ещё может **паниковать** при редких сбоях хранилища — см. [store-startup-failure-semantics.md](store-startup-failure-semantics.md).
- **Отправка / subscribe / refresh:** **`false`** при ошибках валидации, отсутствии адресов или ошибках хранилища; детали обычно в **stderr** через `print_error!`.
- **Status callback** (`lnr_set_status_cb`): асинхронные peer up/down (только связанные топики) и фоновые сбои route/store/send — см. [using-the-api.md](using-the-api.md) и [errors-and-logging.md](errors-and-logging.md). Не заменяет синхронные коды возврата.
- **Очистка данных:** **`clear_stored_messages`** и **`clear_addresses_of_topic`** успешны только когда клиент **не** в running.

Полная матрица (Rust `Client`, `Liner`, C, poison mutex) — в [errors-and-logging.md](errors-and-logging.md).

## Краткая ментальная модель

1. **Хранилище** = общая телефонная книга: топик → TCP-адреса + метаданные для офлайн-очередей.  
2. **Исходный топик** = ваше опубликованное имя и адрес привязки.  
3. **`send_to` / `send_all`** = поиск пиров по строке топика, затем TCP + опциональная персистентность.  
4. **`subscribe`** = какие входящие логические топики принимает ваш колбэк.  
5. **Ошибки** = синхронный **boolean / null** плюс **stderr**; асинхронные peer/route — опционально через **status callback**.
