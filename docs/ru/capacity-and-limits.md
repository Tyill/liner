# Пропускная способность, лимиты и константы настройки

Все числовые константы лежат в **`src/settings.rs`**, если не указано иначе. Это **руководство по размерам** для интеграторов, а не гарантия каждого крайнего случая в коде.

---

## Жёсткий предел: одно кадрированное сообщение по проводу

**`BYTESTREAM_MAX_MESSAGE_SIZE`** (по умолчанию **1 ГиБ**, `1024 * 1024 * 1024`)

- Каждый TCP-кадр начинается с **4 байт big-endian `u32` длины**, затем полезная нагрузка (`bytestream::read_stream`).
- Если объявленная длина **0** или **больше `BYTESTREAM_MAX_MESSAGE_SIZE`**, читатель пишет ошибку в лог и считает поток **закрытым** (ветка `is_shutdown`): сообщение не возвращается.
- Полезная нагрузка выделяется в **mempool** соединения (`mempool.alloc(msg_len)`). Вредоносный или багованный пир с длиной около предела может заставить приёмник **зарезервировать очень много RAM**.

**Практический предел:** считайте **~1 ГиБ на один кадрированный read** настроенным потолком библиотеки; заголовок **`u32`** в принципе допускает до 4 ГиБ, но эта константа — принудительная граница.

Меньшие внутренние I/O-буферы (**`BYTESTREAM_READ_BUFFER_SIZE`** / **`BYTESTREAM_WRITE_BUFFER_SIZE`**, по умолчанию **8 КиБ**) управляют только **чанкингом** чтения/записи, не максимальным размером сообщения.

## Опциональное сжатие полезной нагрузки (zstd)

**`MIN_SIZE_DATA_FOR_COMPRESS_BYTE`** (по умолчанию **1 МиБ**, `1024 * 1024`)

- В **`Message::new`**, если **`data.len() > MIN_SIZE_DATA_FOR_COMPRESS_BYTE`** (строго **больше** 1 МиБ), реализация пробует **`zstd::stream::encode_all`**.
- При успехе по проводу сохраняются **сжатые байты** и выставляется флаг **`COMPRESS`**, чтобы **`get_data`** распаковывал на listener’е.
- Полезные нагрузки **≤ 1 МиБ** уходят **без сжатия** независимо от сжимаемости.

**`DATA_COMPRESS_LEVEL`** (по умолчанию **`0`**) — передаётся в zstd; уровень **`0`** означает «значение по умолчанию zstd» (в коде задокументировано как сейчас близко к уровню **3**).

Сжатие **best-effort**: при ошибке кодировки код пишет в лог и может всё же пометить сообщение сжатым с **пустой** сжатой полезной нагрузкой — вызывающий код должен считать сжатие оптимизацией, а не заменой проверок целостности на уровне приложения.

## Mempool (RAM на соединение)

Mempool — арена в стиле bump на базе **`Vec<Vec<u8>>`** чанками по **`MEMPOOL_CHUNK_SIZE_BYTE`** (по умолчанию **256 КиБ**). Выделения наращивают хранилище целыми чанками по мере необходимости.

| Константа | По умолчанию | Роль |
|-----------|---------------|------|
| **`MEMPOOL_CHUNK_SIZE_BYTE`** | 256 КиБ | Единица роста / выравнивания бэкинга. |
| **`MEMPOOL_MIN_PERCENT_FOR_COMPRESS`** | `0.2` (20%) | При выделении из **фрагментированного свободного пространства** аллокатор может **отказаться** от слияния, если **оставшаяся свободная** доля после alloc **ниже 20%** текущего размера пула — тогда **добавляются новые чанки**. Обмен памяти на менее агрессивное слияние под нагрузкой (имя историческое, не «включить zstd»). |
| **`MEMPOOL_MIN_PERCENT_FOR_RESIZE`** | `0.25` (25%) | Вместе с размером включает **сжатие хвоста** буфера бэкинга, когда в конце сидит крупный выровненный свободный блок. |
| **`MEMPOOL_OVER_SIZE_MB`** | **16** | Бэкинг пула должен превысить **16 МиБ**, прежде чем сработает логика сжатия хвоста (ниже старого **64**, чтобы RSS раньше падал после всплесков). |
| **`MEMPOOL_FREE_COUNT_FOR_RESIZE`** | **4096** | После стольких вызовов **`free`** пул выполняет **`check_free_mem(0)`** для **дефрагментации / слияния** свободных блоков (чаще старого дефолта, чтобы RSS мог падать после всплесков). |

**Что с очень большими сообщениями**

- Построение или приём **большого сообщения** наращивает mempool достаточным числом **чанков по 256 КиБ** (плюс заголовки). **Явного потолка** в mempool кроме **памяти ОС** и **bytestream max** на чтении нет.
- Очень большие одиночные сообщения ⇒ **большой RSS** для mempool этого соединения и при **at-least-once** потенциально крупные записи в очередях **Redis/SQLite**.

## Размеры TCP-буферов (сторона Rust)

| Константа | По умолчанию | Где |
|-----------|---------------|-----|
| **`READ_BUFFER_CAPASITY`** | 64 КиБ | `BufReader` вокруг TCP read в listener. |
| **`WRITE_BUFFER_CAPASITY`** | 64 КиБ | `BufWriter` вокруг TCP write в sender. |

Влияют на пакетирование syscall, не на логический максимум сообщения.

## Короткий чеклист по размерам

1. **На сообщение:** прикладная полезная нагрузка ≤ **1 МиБ** → zstd не пробуется; **> 1 МиБ** → zstd может запуститься (стоимость CPU, меньше по проводу, если данные сжимаются).
2. **На кадрированное TCP-сообщение:** объявленная длина должна быть **≤ `BYTESTREAM_MAX_MESSAGE_SIZE`** (по умолчанию 1 ГиБ), иначе соединение обрывается на этом пути чтения.
3. **RAM:** закладывайте **пик одновременных сообщений × след в mempool** на активное соединение (listener и sender используют mempool для своих рабочих списков). Добавьте запас на **фрагментацию** (аллокатор может держать лишние чанки, когда правило 20% блокирует слияние).
4. **Диск / память Redis:** офлайн-очереди **at-least-once** хранят **закодированные** блобы сообщений; размер ≈ размер по проводу (со сжатием, если оно применялось).
5. **DoS / недоверенные пиры:** заголовок длины недоверен; ограничивайте ожидания на границе сети или на уровне приложения, если 1 ГиБ неприемлем.
